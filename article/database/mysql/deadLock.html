日志大致分为三个部分 ​

TransactionA 第一个事务信息
TransactionB 第二个事务信息
WE ROLL BACK TRANSACTION(2) 最终的处理结果，表示回滚了第二个事务

SessionA ​

SQL2 ​

WAITING FOR THIS LOCK TO BE GRANTED表示这个事务在等待的锁
index PRIMARY of table database.table表示在等待 table表中主键索引上面的锁
lock_mode X locks rec but not gap表示想添加的是一个排它锁（非GAP锁，应该是主键索引）
Record lock 表示是一个记录锁
n_fields 71 表示这个记录是71列
0: len 1: len n:len表示列的值，十六进制
asc 表示 后面跟的是“可打印字符”

SessionB ​

SQL1 ​

HOLDS THE LOCK(S)表示持有锁
index PRIMARY of table mob_online_market.eshop_order_record 表示持有eshop_order_record表中主键索引上面的锁
lock_mode X locks rec but not gap表示持有的是一个排它锁（非GAP锁，应该是主键索引）
Record lock 表示是一个记录锁
n_fields 71 表示这个记录是71列
0: len 1: len n:len表示列的值，十六进制
asc 表示 后面跟的是“可打印字符”\

SQL3 ​

WAITING FOR THIS LOCK TO BE GRANTED表示这个事务在等待的锁
index order_id of table mob_online_market.eshop_order_record 表示在等待 eshop_order_record表中orderId（非主键索引） 索引上面的锁
lock_mode X locks rec but not gap表示想添加的是一个排它锁（非GAP锁，应该是主键索引）
Record lock 表示是一个记录锁
n_fields 2 表示这个记录是2列
0: len 1: len n:len表示列的值，十六进制
asc 表示 后面跟的是“可打印字符”

根据日志分析得出执行的具体流程如下

SessionASessionB
SQL1
SQL2
SQL3

SQL1：根据主键索引更新，对主键加锁
SQL2：根据非主键索引更新，先根据非主键索引加锁，在根据主键加锁，但此时某条记录的已经被SQL1锁在事务持有，等待
SQL3：根据非主键索引更新，先根据非主键索引加锁，此时非主键索引已经被SQL2锁在事务持有，等待。
